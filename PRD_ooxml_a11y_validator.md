# PRD: In-House Deterministic OOXML Accessibility Validator/Remediator ("office-verify")

**Status:** Draft for review
**Owner:** remedy-server engineering
**Repo:** `remedy-server` (this repo)
**Related documents:** `RESEARCH_remedy_server_ADA_refactor.md` (§3.3, §5, §6.2, §9/9.1), `PHASE0_SPEC.md`

---

## 0. Bottom line up front

remedy-server has no formal, rule-based, pass/fail conformance gate for Office files today — `office_acceptance.py` runs 4-6 heuristic checks per format with no rule taxonomy, no level classification, and no VPAT integration. No open-source tool fills this gap either, as best as this session's research could determine (checked against every real candidate found: Microsoft Accessibility Checker, LibreOffice UNO API, `odfvalidator`/`office-o-tron`, Apache POI, DAISY Ace, ISO/IEC 29500 itself, and `Community-Access/accessibility-agents` — six candidates, not an exhaustive market scan). This PRD specifies **office-verify**: an in-house, dependency-light, deterministic OOXML accessibility validator/remediator that (a) reads raw `.docx`/`.pptx`/`.xlsx` ZIP/XML parts directly, (b) applies a Matterhorn-style checkpoint/condition rule taxonomy scoped to what OOXML actually exposes, (c) emits machine-checkable pass/fail verdicts feeding the same `Conformance` / L0–L5 / VPAT 2.x ACR model the PDF side already uses (`compliance_report.py`, `levels.py`), and (d) serves as both an external-file linter and the acceptance gate for remedy-server's own `office_remediator.py` output — the same role `pdf_acceptance.py` + veraPDF play for the PDF path today.

**Drafting/target-approval date: 2026-06-30.** The phased schedule in §9 is anchored to this date; if approval slips materially past 2026-Q3, the Phase 0–4 quarter labels (and specifically Phase 4's 2027-03-31 exit, which already leaves only a 4-week buffer before the April 26, 2027 deadline) must be re-checked against the actual approval date before the plan is treated as binding.

## 1. Problem statement (grounded in this session's verified research)

### 1.1 No veraPDF-equivalent exists for OOXML — confirmed, not assumed

The PDF side of this engine has a real conformance backstop: `validate_with_verapdf()` in `pdf_acceptance.py` shells out to the veraPDF binary, parses its PDF/UA-1 rule-failure XML, and gates `PDFAcceptanceResult.passed` on it. No equivalent exists for Office formats, and — critically — **none was found among the candidates checked this session**, which covered every real candidate this research pass could identify but is not a claim of exhaustive market coverage. This was checked against primary sources this session, not assumed:

| Candidate | What it actually is | Why it doesn't close the gap |
|---|---|---|
| Microsoft Accessibility Checker | UI feature in Word/PowerPoint/Excel | No API, no CLI, no OSS engine — cannot be scripted or run headlessly |
| LibreOffice UNO accessibility API | Exposes the **runtime AT (assistive-technology) tree of a live, open document** | Built for screen readers driving a running LibreOffice instance, not for statically linting a closed file. Wrong object model for a CI-style checker. |
| `odfvalidator` / `office-o-tron` | Real, JAR-based, scriptable validators | They check **XML schema/structural conformance against ISO/IEC 29500/ODF**, not accessibility semantics. A `.docx` can be 100% schema-valid and have zero alt text, no headings, and scrambled reading order. |
| Apache POI | General-purpose read/write library for OOXML | No accessibility module of any kind |
| DAISY Consortium's Ace | Deterministic, real, well-built | **EPUB-only.** No OOXML equivalent exists and no plans were found. |
| ISO/IEC 29500 | Defines the file format itself | Format-only. Unlike PDF, which has PDF/UA-1 as a WCAG-mapped conformance profile on top of the base ISO 32000 spec, **no equivalent WCAG-mapped accessibility conformance profile exists for OOXML to implement.** There is no "OOXML/UA-1" to target. |

### 1.2 The closest artifact — a reference pattern, not a dependency

`Community-Access/accessibility-agents` (MIT, v6.0.0, 348 stars, actively maintained) contains `.github/scripts/office-a11y-scan.mjs` (307 lines). Read the actual source, not just the docs, this session:

- **It is genuinely deterministic**: pure Node.js, `node:fs`/`node:zlib` only, zero npm dependencies, a hand-rolled ZIP central-directory parser plus regex-based XML part extraction. Runs headlessly, exits 1 on error, emits SARIF 2.1.0 and GitHub Actions annotations.
- **It implements far less than advertised**: ~10-12 checks total across docx/pptx/xlsx combined, versus 46 claimed in its own documentation — a confirmed documentation/code mismatch, not a rounding difference.
- **It has zero remediation capability.** "Fixing" in that repo is live LLM code-generation with a human in the loop reviewing a diff — not a deterministic library function that mutates the file. This is the opposite of what remedy-server's `office_remediator.py` already does today (deterministic, non-LLM structural fixes, with an *optional* vision-LLM call only for alt-text generation).

**Conclusion carried into this PRD:** the ZIP-central-directory + XML-regex-extraction + SARIF/CI-annotation architecture is a reasonable pattern to reference for "how do you deterministically parse OOXML without a heavy dependency," but no code is copied and no dependency is taken — remedy-server already has stronger primitives (`python-docx`, `python-pptx`, `openpyxl`, plus its own raw-ZIP `ElementTree` parsing in `behavioral_proxies/office/_ooxml.py` and `_alt_text.py`) that should be extended rather than replaced.

### 1.3 remedy-server's own current state — read from the actual files, not inferred

Contrary to a naive "there's nothing yet" assumption, three tiers of Office capability already exist in this repo, at very different levels of formality:

1. **Remediation** — `office_remediator.py` (414 lines): deterministic python-docx/python-pptx/openpyxl mutation. Sets title/language metadata, infers heading candidates via layout heuristics (`_is_heading_like_paragraph`), marks `w:tblHeader` on first table rows, sets `docPr`/`cNvPr` `descr`/`title` alt-text attributes (optionally via an injected vision LLM, degrading gracefully to a filename-derived default), and for xlsx enables freeze panes / autofilter / repeating print rows. No validation gate precedes or follows this — it is remediation with no acceptance contract enforcing it worked.
2. **Ad-hoc acceptance** — `office_acceptance.py` (465 lines): `run_office_checker()` implements exactly **5 checks for docx** (`docx-title`, `docx-language`, `docx-headings`, `docx-table-headers`, `docx-alt-text`), **4 for pptx** (`pptx-title`, `pptx-language`, `pptx-slide-titles`, `pptx-alt-text`), **3 for xlsx** (`xlsx-title`, `xlsx-language`, `xlsx-header-behaviors`). Each is a single boolean pass/fail with no rule-ID taxonomy, no checkpoint/condition hierarchy, no severity levels beyond implicit pass/fail, and — critically — **no formal specification these checks trace back to** (unlike the PDF side's Matterhorn/PDF-UA-1 lineage). `OfficeAcceptanceResult.passed` is wired into `backend/app/office_routes.py`'s synchronous `/v1/office/check` endpoint but that endpoint is not gated on any external, versioned rule spec.
3. **Deep quality/behavioral infrastructure — already extensive, larger than the ticket assumed.** `behavioral_proxies/office/{docx,pptx,xlsx}/` and `quality_judges/office/{docx,pptx,xlsx}/` implement **7 quality dimensions per format** (`alt_text`, `reading_order`/`slide_reading_order`, `heading_semantics`/`slide_title`, `table_structure`, `link_text`, `decorative`, `complex_content`, plus xlsx's `sheet_organization`) via `quality_judges/office/audit.py`'s `audit_office_quality()`. These are LLM-assisted judges layered on top of the same raw ZIP/XML parsing (`_ooxml.py`, `_alt_text.py`) — they are **not** deterministic pass/fail rule checks in the veraPDF sense; they produce 0.0–1.0 confidence scores gated at an 0.80 threshold (`alt_text_substitution.py`), explicitly the wrong tool for a machine-checkable conformance gate but the *right* place to route the non-machine-checkable half of the taxonomy (§6 below). **office-verify must not duplicate this layer — it must sit underneath it as the deterministic gate the quality layer currently lacks**, and the two together (deterministic rule engine + LLM quality judges) form the OOXML analogue of the PDF side's (`veraPDF` + vision-based quality judges) split.

**The actual gap, precisely stated:** there is no versioned, rule-ID-addressable, Matterhorn-style checkpoint taxonomy behind `office_acceptance.py`'s 12 checks; no L0–L5 classifier for Office documents (`levels.py` is PDF-only — `classify_level()` calls `pikepdf.open()` directly and asserts a PDF-shaped `StructureProbe`); no VPAT ACR generation for Office (`compliance_report.py`'s `WCAG_MAPPING` list and `_determine_conformance()` are PDF-only, importing `pdf_checker.CheckResult` and `tag_tree_reader.ScreenReaderIssue` types that don't exist for Office); and `office_acceptance.py` is never used as a formal *acceptance gate* for `office_remediator.py`'s own output the way `pdf_acceptance.py` gates PDF remediation output today (no `office_remediator.py` call site invokes `evaluate_office_acceptance()` post-remediation; only the standalone `/v1/office/check` API route does).

## 2. Goals

1. **G1 — Formal rule taxonomy.** Define a versioned, numbered rule catalog for docx/pptx/xlsx modeled on Matterhorn's checkpoint/condition structure, each rule naming the exact OOXML XML part(s)/element(s)/attribute(s) it reads, with a stable `rule_id` (e.g. `OOXML-DOCX-1.1.1-a`) so results are diffable across engine versions.
2. **G2 — Deterministic engine, zero required external dependencies.** Every rule must be answerable by parsing already-open `python-docx`/`python-pptx`/`openpyxl` object models or, where those libraries don't expose a needed field, by direct `zipfile` + `xml.etree.ElementTree` part reads (the same pattern already used in `behavioral_proxies/office/_ooxml.py`). No LLM call may be required for a rule to return Pass/Fail — LLM-assisted quality judging remains a separate, already-existing layer (§6).
3. **G3 — Machine-checkable pass/fail verdicts**, not scores, for every rule in the deterministic tier — mirroring `CheckResult.status ∈ {"Passed","Failed","Manual Check Needed"}` on the PDF side.
4. **G4 — Feed the existing L0–L5 + VPAT ACR model.** Extend `levels.py` and `compliance_report.py` (or add Office-specific siblings with the same contract) so Office documents get a level classification and a VPAT 2.x ACR exactly analogous to PDFs, not a parallel incompatible format.
5. **G5 — Serve as the acceptance gate for remedy-server's own remediation output.** `office_remediator.py`'s output must be run back through the new validator before being considered "done," exactly as `pdf_acceptance.py` gates `pdf_fixer.py`/rebuild output today.
6. **G6 — Deterministic auto-remediation where mechanically safe.** For rules whose fix is unambiguous (e.g., set `docProps/core.xml` language, mark `w:tblHeader`), the validator's failure output must be directly actionable by `office_remediator.py` (already true for several checks) — and the PRD should close remaining gaps (e.g., pptx reading-order in the selection pane, xlsx merged-cell detection).
7. **G7 — Open-source, in-house, no new runtime dependency** beyond what's already vendored (`python-docx`, `python-pptx`, `openpyxl`, stdlib `zipfile`/`xml.etree`).
8. **G8 — Ship docx first, on a schedule compatible with the April 26, 2027 LACCD ADA deadline**, per the phased plan in §9.

## Non-goals

- **Not a general OOXML schema validator.** office-verify does not re-implement what `odfvalidator`/`office-o-tron` already do (ISO/IEC 29500 structural conformance). It assumes the file is schema-valid enough to open with python-docx/pptx/openpyxl (openability is a precondition, already checked by `validate_office_package()`) and focuses exclusively on accessibility semantics.
- **Not a replacement for the LLM quality-judge layer.** `quality_judges/office/*` and `behavioral_proxies/office/*` continue to own alt-text *quality* (is this alt text substitutive, not just present), reading-order *comprehension*, and other judgment calls that are not machine-decidable from XML alone. office-verify owns *presence/structure*, they own *quality/semantics*.
- **Not a live-document AT-tree walker** (that's what LibreOffice UNO does and was explicitly ruled out as the wrong model — §1.1).
- **Not a legacy-format (`.doc`/`.ppt`/`.xls`) engine.** Those require a LibreOffice-headless conversion front-end per `RESEARCH_remedy_server_ADA_refactor.md` §6.2 gap #5; out of scope here, tracked as a dependency (§10).
- **Not going to emit an "ADA compliant"/L5 verdict from automation.** Per the L0-L5 invariant already enforced in `levels.py` (`classify_level` asserts `level in {"L0"…"L4"}` and never emits L5), the Office-side classifier must carry the identical invariant. Emitting a fully-automated "compliant" verdict is the *Payan v. LACCD* "deliberate indifference" legal trap the PDF side was explicitly built to avoid (`RESEARCH_remedy_server_ADA_refactor.md` §1.4-1.5).
- **Not rebuilding `office_remediator.py`'s mutation logic.** This PRD adds a validation/classification/reporting layer; remediation-side gaps it surfaces (e.g., no merged-cell splitting) are follow-on tickets, not part of this PRD's deliverable.

## 3. Fit with the existing Remediation-Level taxonomy and VPAT ACR model

`RESEARCH_remedy_server_ADA_refactor.md` §5 defines L0–L5 + L-REPLACE as PDF-native gates (`/StructTreeRoot`, veraPDF PDF/UA-1, `/Lang`, `DisplayDocTitle`, PDF/UA XMP id). None of these PDF-specific catalog objects exist in OOXML. office-verify must translate the *intent* of each gate into OOXML-native equivalents so a District-wide burndown report (`levels.summarize_levels()`) can show PDF and Office documents on the same L0–L5 axis without conflating unrelated internals.

### 3.1 Proposed Office level gates (`office_levels.py`, new module, same contract as `levels.py`)

| Level | PDF gate (existing, `levels.py`) | Office-native equivalent (new) |
|---|---|---|
| **L0** Inaccessible | No extractable text layer | No extractable text in any `w:t`/`a:t`/cell value across the document body — i.e., an image-only scan wrapped in an Office container, or an empty shell. Rare but occurs (scanned page embedded as a single full-slide image in pptx). |
| **L1** Text, untagged | Text present, no `/StructTreeRoot` | Text present, but **zero heading-style paragraphs/slide-titles** and **zero explicit structural signal** — i.e., `office_acceptance.py`'s `docx-headings`/`pptx-slide-titles` both fail. OOXML has no separate "tag tree" the way PDF does; heading styles + `w:tblHeader` + `docPr` alt-text attributes *are* the structure layer, so "untagged" = none of these present. |
| **L2** Auto-tagged, unverified | `/Marked true` + `/StructTreeRoot`, fails veraPDF | Some structural signal present (headings exist, or table headers exist, or alt text exists) but the deterministic rule set (§4) still has ≥1 machine-checkable **Fail** (not just "Manual Check Needed"). This is where `office_remediator.py`'s current heuristic output usually lands today — real structure was added, but not verified against a rule catalog. |
| **L3** Machine-verified | veraPDF clean + `/Lang` + `DisplayDocTitle` + PDF/UA id | **All deterministic rules Pass** (§4's full catalog) — title metadata set, language set, headings present with valid hierarchy, all images have non-empty alt attributes, all tables expose header rows, no OOXML-detectable color-only-meaning patterns. This is office-verify's actual "veraPDF-equivalent" gate. |
| **L4** WCAG 2.1 AA engine-complete | L3 + quality layer (`quality_result.overall_pass`) | L3 + `audit_office_quality().overall_pass` (the existing `quality_judges/office/audit.py` ensemble) — i.e., deterministic rules clean **and** the LLM-assisted alt-text-substitutiveness / reading-order / heading-semantics judges all score ≥0.80. This wires the two existing-but-disconnected layers (§1.3 point 2 and 3) into one ladder, mirroring how PDF's L4 already combines `verapdf_result` + `quality_result`. |
| **L5** Human-validated | Never machine-assigned (invariant) | Never machine-assigned (same invariant, enforced by an identical `assert level in {"L0"..."L4"}` in `office_levels.classify_level()`). |
| **L-REPLACE** | Authoritative District PDF match found | Unchanged — orthogonal to file format; the existing campus↔District matching pipeline output applies identically whether the matched pair is PDF or Office. |

### 3.2 VPAT ACR integration

`compliance_report.py`'s `WCAG_MAPPING: list[WCAGCriterion]` and `_determine_conformance()` are PDF-shaped (they import `CheckResult`/`ScreenReaderIssue` and read `PDFAcceptanceResult`-only fields like `verapdf_result`). Two structurally-consistent options, both preserving one shared report contract:

**`DocumentReport` is not actually format-agnostic today — this is a real technical-design gap, not a solved problem.** Reading the dataclass directly (`compliance_report.py` lines 340-379): alongside genuinely shared fields (`document_name`, `conformance`, `wcag_results`, `issues`), it carries *required*, non-defaulted PDF-only fields — `remediated_pages` (an xlsx has no "pages"), `tag_count` (no `/StructTreeRoot` equivalent to count), `verapdf_checked`/`verapdf_passed`/`verapdf_violations` (no veraPDF run for Office), and a `visual_diff` dict explicitly documented as "page-by-page pixel comparison." It also exposes a `source_font_only` property that inline-imports `PDFAcceptanceResult` and calls `PDFAcceptanceResult._is_source_font_limitation()` directly — a hard PDF-only code path, not just an unused field. None of these have a natural OOXML meaning, so "reused unchanged" is not achievable without either populating dummy/zero values for fields that mean nothing for Office, or forking the dataclass — which is precisely the coupling problem used below to reject Option B.

- **Option A (recommended, revised): parallel `office_compliance_report.py`** that imports a new `OfficeWCAGCriterion` mapping (same `id`/`name`/`level`/`rule_ids` shape) pointing at office-verify's `rule_id`s instead of PDF `CheckResult.rule_id`s, and a `_determine_office_conformance()` mirroring `_determine_conformance()`'s three-tier `Conformance.CONFORMANT/PARTIALLY/NOT_CONFORMANT` logic against `OfficeAcceptanceResult`. Given the PDF-only required fields above, `DocumentReport` itself must be extended — not reused byte-for-byte — by making `remediated_pages`, `tag_count`, `verapdf_checked`, `verapdf_passed`, `verapdf_violations`, and `visual_diff` `Optional`/defaulted rather than required, and moving `source_font_only`'s `PDFAcceptanceResult` import/call behind a guard that only fires when `verapdf_checked` is true. An `OfficeDocumentReport` subclass (or a shared base dataclass with a thin PDF-specific and Office-specific subclass each) is an acceptable alternative if widening the existing dataclass's optionality is judged too risky for the PDF path. Either way, FR4 must state explicitly that this is "extend with optional PDF-only fields," not "reuse unchanged" — the latter does not survive contact with the real dataclass.
- **Option B: extend `compliance_report.py` in place** with `if file_type in OFFICE_TYPES: ...` branches. Rejected — this couples PDF-only imports (`pikepdf`, `pdf_checker`) into a module that must also work when neither is relevant, and every future PDF-report change risks an Office regression and vice versa. Option A's duplication cost is a new, comparably-sized sibling module — `compliance_report.py` itself is 1217 lines, so `office_compliance_report.py` should be estimated as a meaningful fraction of that (its own `WCAG_MAPPING`, `_determine_office_conformance()`, `_render_html()` analogue, serialization/deserialization glue), not a small shim — that cost is still worth the isolation, matching how `office_acceptance.py` (465 lines) was already split out from `pdf_acceptance.py` rather than merged into it. A concrete line-count estimate for `office_compliance_report.py` should be produced during Phase 0 scoping (§9) once the extended-`DocumentReport` shape is settled, rather than guessed here.

The 23-row `WCAG_MAPPING` table in `compliance_report.py` maps 1:1 conceptually onto Office; concretely, rows relevant to Office (with office-verify `rule_id`s substituted) are: **1.1.1** Non-text Content → alt-text rules; **1.3.1** Info and Relationships → heading/table-header/list rules; **2.4.2** Page Titled → title-metadata rule; **2.4.6** Headings and Labels → heading-hierarchy rule; **3.1.1** Language of Page → language-metadata rule; **4.1.2** Name, Role, Value → form-field-label rules (docx content controls, xlsx data-validation prompts). Criteria with no OOXML-checkable signal (e.g., **1.3.2** Meaningful Sequence for pptx reading order beyond title-shape-first ordering) stay `N/A` under automation and route to `needs_human`, exactly like the PDF side's `_LLM_HANDLED_CHECKS`/`needs_human` split in `levels._build_needs_human()`.

## 4. Rule taxonomy (Matterhorn-style checkpoint/condition structure, scoped to what OOXML exposes)

Modeled on Matterhorn Protocol's `Checkpoint.Condition` numbering (e.g. `01-001`), but renamed to avoid implying PDF/UA lineage: **`OOXML-<FORMAT>-<Checkpoint>.<Condition>`**. Each rule below states the exact XML part(s)/element(s)/attribute(s) it reads — no rule is defined against an abstraction that doesn't map to a concrete OOXML location.

### 4.1 DOCX checkpoints

| Rule ID | Checkpoint | Condition (machine test) | OOXML part/element read |
|---|---|---|---|
| `OOXML-DOCX-1.1` | Document metadata | Title present in package core properties | `docProps/core.xml` → `dc:title` (already read via `doc.core_properties.title` in `office_acceptance._check_docx`) |
| `OOXML-DOCX-1.2` | Document metadata | Language present in package core properties | `docProps/core.xml` → `dc:language` (`doc.core_properties.language`) |
| `OOXML-DOCX-2.1` | Heading structure | ≥1 paragraph carries a `Heading`/`Title` style or `w:outlineLvl` | `word/document.xml` → `w:p/w:pPr/w:pStyle/@w:val` matching `Heading*`/`Title`, or `w:p/w:pPr/w:outlineLvl/@w:val` (mirrors `_docx_paragraph_has_heading_structure`) |
| `OOXML-DOCX-2.2` | Heading structure | Heading levels do not skip (H1→H3 without H2) | Same elements as 2.1; walk paragraph order, diff consecutive `w:outlineLvl` values |
| `OOXML-DOCX-2.3` | Heading structure | Document does not open on a body paragraph before any heading/title (a "no orphan intro text" check) | Same; first non-empty paragraph's style |
| `OOXML-DOCX-3.1` | Images | Every inline/anchored image exposes non-empty `descr` or `title` | `word/document.xml` → `w:drawing//wp:inline/wp:docPr/@descr` and `@title` (and `wp:anchor/wp:docPr` for floating images — **not currently read by `office_acceptance._check_docx`, which only iterates `doc.inline_shapes`; anchored/floating images are a real gap to close**) |
| `OOXML-DOCX-3.2` | Images | Alt text is not a filename/placeholder pattern (`img\d+\.(png\|jpe?g)`, `image1`, empty after trim) | Same attributes; regex/heuristic on the `descr` value — deterministic pattern match only, not semantic judgment (semantic judgment stays with `quality_judges/office/docx/alt_text_judge.py`) |
| `OOXML-DOCX-4.1` | Tables | First row of every `w:tbl` is marked as a repeating header | `word/document.xml` → `w:tbl/w:tr[1]/w:trPr/w:tblHeader` (mirrors existing `docx-table-headers` check) |
| `OOXML-DOCX-4.2` | Tables | No table uses merged cells (`w:gridSpan`/`w:vMerge`) in the header row | `w:tbl/w:tr[1]//w:tcPr/w:gridSpan`, `w:vMerge` — **new rule**, not in current `office_acceptance.py` |
| `OOXML-DOCX-5.1` | Lists | Bulleted/numbered visual patterns (paragraphs starting with `-`/`•`/`1.`) are backed by real `w:numPr`, not manual characters | `word/document.xml` → `w:p/w:pPr/w:numPr` vs. leading-character text heuristic — **new rule**; requires deterministic text-pattern check (e.g., regex `^[•\-*]\s` or `^\d+[.)]\s`) cross-referenced against absence of `w:numPr` |
| `OOXML-DOCX-6.1` | Hyperlinks | Hyperlink display text is not a bare URL or generic "click here"/"read more" | `word/document.xml` → `w:hyperlink` run text vs. `r:id` target; deterministic pattern match on display text only (substitutive-quality judgment stays with `quality_judges/office/docx/link_text_judge.py`) |
| `OOXML-DOCX-7.1` | Color-only meaning | No paragraph/run relies solely on a color run-property change adjacent to referential text like "shown in red" | `word/document.xml` → `w:r/w:rPr/w:color` co-occurring with regex-matched color-reference phrases — **best-effort deterministic pattern rule**, flagged for human review rather than hard-failed given false-positive risk |

### 4.2 PPTX checkpoints

| Rule ID | Checkpoint | Condition | OOXML part/element read |
|---|---|---|---|
| `OOXML-PPTX-1.1` | Presentation metadata | Title present | `docProps/core.xml` → `dc:title` |
| `OOXML-PPTX-1.2` | Presentation metadata | Language present | `docProps/core.xml` → `dc:language` |
| `OOXML-PPTX-2.1` | Slide titles | Every slide has a non-empty title placeholder | `ppt/slides/slideN.xml` → `p:sp[p:nvSpPr/p:nvPr/p:ph/@type='title']//a:t` (mirrors existing `pptx-slide-titles`, which currently falls back to "first non-empty text frame" when no title placeholder exists — **should be tightened to require a real title placeholder, not any text frame**, since a fallback text frame is not exposed to a screen reader as a slide title) |
| `OOXML-PPTX-2.2` | Slide titles | Slide titles are unique across the deck (no duplicate/generic "Slide 1" pattern) | Same; cross-slide comparison — **new rule**, addresses `RESEARCH_remedy_server_ADA_refactor.md` §3.3's explicit "unique slide titles" requirement not currently checked |
| `OOXML-PPTX-3.1` | Images | Every `p:pic` exposes non-empty `descr` or `title` | `ppt/slides/slideN.xml` → `p:pic/p:nvPicPr/p:cNvPr/@descr`, `@title` (mirrors existing `pptx-alt-text`) |
| `OOXML-PPTX-3.2` | Images | Decorative images are explicitly marked (`<a:extLst>` decorative extension present) rather than given empty/filler alt | `ppt/slides/slideN.xml` → `p:cNvPr/a:extLst/a:ext` decorative flag (OOXML 2016+ decorative-image extension) — **new rule, external-spec-sourced, not yet verified against this codebase**: unlike `3.1` (which mirrors the existing, code-verified `pptx-alt-text` check), no `decorative_judge.py`/`_decorative.py` code path in this repo was found reading this extension, so this row is asserted from external OOXML spec knowledge only. Treat with the same "best-effort" hedge as the color-only-meaning rules until a Phase 2 spike confirms python-pptx/`ElementTree` can actually read this element reliably across PowerPoint-produced files; without it, a "Passed" alt-text check on a genuinely decorative image can only be verified by explicit decorative marking, not by absence of `descr` |
| `OOXML-PPTX-4.1` | Reading order | Selection-pane / tab order (`p:spTree` child element order) places the title shape first | `ppt/slides/slideN.xml` → `p:cSld/p:spTree` child ordering vs. placeholder type — deterministic structural check; **does not** replace `slide_reading_order_comprehension.py`'s LLM-assisted semantic check, it only catches the mechanical case where the title shape isn't even first in XML order |
| `OOXML-PPTX-5.1` | Tables | Table first row marked as header | `ppt/slides/slideN.xml` → `a:tbl/a:tr[1]` + `a:tblPr/@firstRow="1"` |

### 4.4 Common structure of every rule (implementation contract)

Every rule is implemented as a small pure function with the signature `(package: zipfile.ZipFile | python-docx Document | python-pptx Presentation | openpyxl Workbook) -> RuleResult`, where `RuleResult` mirrors `office_acceptance.OfficeCheckResult` (`rule_id`, `description`, `status: Literal["Passed","Failed","Manual Check Needed"]`, `details: list[str]`, `fixable: bool`) — no new result shape is introduced; the existing dataclass is extended with a formal `rule_id` catalog and checkpoint grouping metadata (`checkpoint`, `wcag_ref`) rather than replaced.

## 5. Machine-checkable vs. human-review split (reasoned estimate)

The PDF side's Matterhorn-derived split is **89 machine-checkable / 45 human-only / 2 undetermined conditions (136 total, across 31 checkpoints)**, cited in `RESEARCH_remedy_server_ADA_refactor.md` §3 ("31 checkpoints / 136 failure conditions... 89 are machine-checkable, 45 require a human, 2 are undetermined — a minor variant split of 87/47/2 appears in some sources"), and restated consistently at §4 ("≈89/136 Matterhorn conditions" machine-checkable, "≈45/136" requiring human judgment). The 2-condition "undetermined" bucket is a source-level ambiguity in the Matterhorn literature itself (not this PRD's own uncertainty) and is not further decomposed here; it is small enough (1.5% of 136) not to change the ~66%/34% machine/human framing the RESEARCH doc draws from it, and no attempt is made below to force an OOXML analogue of that specific bucket. OOXML has no equivalent published taxonomy to cite a count from, so the split below is a reasoned estimate built from the rule catalog in §4 plus the same reasoning Matterhorn used (structural presence is machine-checkable; semantic correctness is not):

- **Rule catalog size (§4):** 8 docx + 8 pptx + 8 xlsx = **24 checkpoints** defined above. This is smaller than PDF's 136 because OOXML exposes far less structural machinery than the PDF tag tree (no `/StructTreeRoot` hierarchy, no `ParentTree`, no `MCID`-to-content mapping) — there is simply less to check mechanically.
- **Fully machine-checkable (Pass/Fail, no ambiguity):** metadata presence (title/language: 2 per format = 6), structural presence (headings exist, table headers marked, alt attribute non-empty, sheet names non-default, no merged cells: ~4 per format = 12), and duplicate/uniqueness checks (slide titles unique: 1) → **≈19 of 24 checkpoints**, i.e. **~79%** machine-checkable outright.
- **Pattern-based / best-effort (machine-flaggable but with real false-positive risk, so routed to "Manual Check Needed" rather than a hard Fail):** color-only-meaning heuristics (3, one per format), manual-bullet-vs-real-list detection (1), decorative-image-marking absence (1) → **5 of 24**, **~21%**.
- **Genuinely non-machine-checkable (owned by the LLM quality-judge layer, never a hard rule):** alt-text *substitutiveness* (is "image1.png" replaced by something actually descriptive, vs. merely non-empty — owned by `alt_text_judge.py`), reading-order *comprehension* (does the actual visual/logical flow match navigation order, vs. merely "title shape is XML-first" — owned by `reading_order_judge.py`/`slide_reading_order_judge.py`), heading *semantic correctness* (is this really a heading, vs. bold body text styled to look like one — owned by `heading_semantics_judge.py`), table *semantic* structure (owned by `table_structure_judge.py`), link-text *quality* (owned by `link_text_judge.py`), decorative-image *correctness* (owned by `decorative_judge.py`), and OCR/embedded-object content accuracy. These map roughly 1:1 onto Matterhorn's **45-condition human-only category** (plus the 2-condition undetermined bucket, not separately modeled — see above) cited for PDF, scaled down for OOXML's smaller structural surface: **estimated 15-20 additional conditions**, already served by the 7-dimension LLM quality-judge ensemble in `quality_judges/office/audit.py` rather than needing new machine rules.

**Total estimated OOXML taxonomy: ~24 deterministic rule-catalog entries (this PRD's scope) + ~15-20 judgment-only conditions (already covered by the existing quality-judge layer) ≈ 40 total conditions** — smaller in absolute count than PDF's 136 because OOXML's native structure (styles, `docPr`, `w:tblHeader`) is coarser-grained than a full PDF tag tree, but with a similar ~75-80%/20-25% machine/human split, consistent with Matterhorn's ~65%/33%/1.5% split for PDF (89/45/2 of 136) — OOXML skews slightly more machine-checkable because its structural signals (a style name, a boolean `tblHeader` flag) are binary XML facts rather than requiring tree-walk inference the way PDF `StructElem` parent/child relationships do.

## 6. Functional requirements

- **FR1.** A new module `src/project_remedy/office_checker.py` (naming parallel to `pdf_checker.py`) implements the full §4 rule catalog as a `OfficeAccessibilityChecker` class with a `run_all() -> OfficeCheckReport` method, replacing/extending `office_acceptance.run_office_checker()`'s current 5/4/3-check implementation. `office_acceptance.py`'s existing `OfficeCheckResult`/`OfficeCheckReport` dataclasses are reused unchanged as the result contract — only the rule catalog behind `_check_docx`/`_check_pptx`/`_check_xlsx` is deepened.
- **FR2.** Every rule must declare, in code, the exact XML part path(s) and element/attribute local names it reads (as a docstring or a structured `xml_refs: tuple[str,...]` field on the rule), so the taxonomy is self-documenting and auditable — this is what makes the catalog "grounded in concrete OOXML," not aspirational.
- **FR3.** `office_levels.py` (new, parallel to `levels.py`) implements `classify_level(acceptance: OfficeAcceptanceResult, probe: OfficeStructureProbe) -> LevelResult` using the identical `LevelResult` dataclass already defined in `levels.py` (import and reuse it — do not redefine), with the OOXML-native gates from §3.1. Must carry the identical `assert level in {"L0","L1","L2","L3","L4"}` invariant.
- **FR4.** `office_compliance_report.py` (new, parallel to `compliance_report.py`) implements `generate_office_document_report()` returning the shared `DocumentReport` contract defined in `compliance_report.py`, **extended with the PDF-only fields (`remediated_pages`, `tag_count`, `verapdf_checked`, `verapdf_passed`, `verapdf_violations`, `visual_diff`) made `Optional`/defaulted rather than required**, and with `source_font_only`'s `PDFAcceptanceResult` call guarded to only fire when `verapdf_checked` is true (see §3.2's revised Option A) — or, if widening the existing dataclass is judged too risky, a shared base dataclass with a thin `OfficeDocumentReport` subclass. This is *not* a zero-cost reuse; it is a deliberate, minimal-surface-area extension so a single HTML/JSON report renderer serves both PDF and Office documents, differing only in the `WCAG_MAPPING` table and conformance-determination function used internally.
- **FR5.** `evaluate_office_acceptance()` in `office_acceptance.py` must gain an explicit call site inside `office_remediator.py`'s `remediate()` (or its call sites in `backend/app/office_routes.py`'s remediate flow) so that Office remediation output is validated before being reported as done — closing the gap noted in §1.3 that `evaluate_office_acceptance()` today is only reachable via the standalone `/v1/office/check` route, never automatically after remediation.
- **FR6.** Rules marked `fixable: True` in the catalog must correspond to an actual, already-implemented or newly-added mutation in `office_remediator.py`. Concretely: `OOXML-DOCX-4.2` (merged header cells), `OOXML-XLSX-2.1` (default sheet names), `OOXML-XLSX-4.1` (merged cells), and `OOXML-PPTX-2.2` (duplicate slide titles) are new rules in this PRD with **no corresponding remediator support today** — closing that gap is in scope for Phase 2/3 (§9), tracked explicitly rather than left as a silent "Failed, fixable=True" that the remediator can't actually act on.
- **FR7.** A CLI entry point `src/project_remedy/cli_office.py` (parallel to the existing `cli_pdf.py`) exposing `check`, `classify-level`, and `report` subcommands, so office-verify is independently scriptable/CI-usable the same way the PDF side is — this is also the natural place to add SARIF output (referencing, not copying, `office-a11y-scan.mjs`'s SARIF 2.1.0 pattern) for GitHub Actions integration on the District's own document repos if desired later.
- **FR8.** Legacy format handling (`.doc`/`.ppt`/`.xls`) is explicitly out of scope for the rule engine itself (Non-Goals), but `cli_office.py`/`office_acceptance.py` must fail closed with a clear "unsupported — requires OOXML conversion first" error rather than silently mis-parsing an OLE2 file as a ZIP (the existing `_OLE2_MAGIC` check in `office_routes.py`'s `_stage_office()` already guards the upload path; the CLI needs the same guard).

## 7. Non-functional requirements

- **NFR1 (determinism).** Given the same input file bytes, `OfficeAccessibilityChecker.run_all()` must produce byte-identical `OfficeCheckReport` output across runs and across machines — no network calls, no LLM calls, no wall-clock-dependent values in the deterministic tier. (LLM-assisted quality judging remains explicitly non-deterministic and is not part of this tier's contract.)
- **NFR2 (performance).** Full rule-catalog evaluation on a single document must complete in well under 1 second for typical LACCD district forms (single-digit-MB docx/xlsx, <50 slides for pptx) — this is a pure in-memory ZIP+XML parse, not a rendering or OCR operation, and should be 1-2 orders of magnitude faster than the PDF checker's veraPDF subprocess call.
- **NFR3 (no new runtime dependencies).** Implementation uses only `python-docx`, `python-pptx`, `openpyxl` (already vendored per `office_remediator.py`'s imports) plus stdlib `zipfile`/`xml.etree.ElementTree` (already used in `behavioral_proxies/office/_ooxml.py`). No new PyPI package, no new binary/JAR dependency (unlike the PDF side's veraPDF subprocess).
- **NFR4 (backward compatibility).** `office_acceptance.OfficeCheckResult`/`OfficeCheckReport`/`OfficeAcceptanceResult` dataclass shapes are not broken — existing call sites (`backend/app/office_routes.py`, `behavioral_proxies/office/_checks.py`, `quality_judges/office/_heuristics.py`, `quality_judges/office/audit.py`) must continue to function unmodified against the deepened rule catalog, since they only depend on the dataclass shape, not the specific 12-check implementation.
- **NFR5 (testability).** Every rule needs a unit test with a minimal synthetic OOXML fixture (a docx/pptx/xlsx built in-memory via `python-docx`/`python-pptx`/`openpyxl` in the test itself, not a checked-in binary blob) exercising both the Pass and Fail path — there is currently **zero test coverage for Office** anywhere in `tests/` (confirmed: `find tests -iname "*office*"` returns nothing), so this is a from-scratch test suite, not an extension of one.
- **NFR6 (auditability).** Every rule's Fail output must cite the specific XML part and, where feasible, the specific element index/slide number/sheet name — mirroring how `office_acceptance.py`'s existing `details` field already reports counts (e.g. `"3 table(s) missing header rows"`); this PRD requires location-specificity to go further where cheap (e.g., which sheet, which slide number) since a District reviewer or the human-review queue (future work) needs to jump to the exact location, not just a count.

## 8. Success metrics / acceptance criteria

All must be machine-checkable via `pytest` in CI, not manually eyeballed:

1. **AC1.** `pytest tests/unit/test_office_checker.py` (new) achieves ≥95% branch coverage of `office_checker.py`'s rule functions, with every rule in §4 having at least one synthetic-fixture Pass test and one Fail test.
2. **AC2.** Running `office-verify check` (via `cli_office.py`) against a known-bad synthetic docx (no title, no language, no headings, images without alt, no table headers) returns exactly the expected set of `rule_id`s as `Failed` — a golden-file regression test, `tests/unit/test_office_checker_golden.py`.
3. **AC3.** `office_levels.classify_level()` never returns `"L5"` under any input — enforced by a property-based/fuzz test feeding it randomized `OfficeAcceptanceResult`/`OfficeStructureProbe` combinations (mirrors how the PDF-side invariant would be tested, extending the same pattern).
4. **AC4.** Round-tripping a document through `office_remediator.remediate()` then `office_checker.run_all()` on the *output* strictly reduces the Failed-rule count versus running the checker on the *input* — i.e., remediation demonstrably improves the machine-checkable score, for a corpus of ≥20 real LACCD sample docx/pptx/xlsx files. This is the FR5 acceptance-gate loop made measurable.
5. **AC5.** `office_compliance_report.generate_office_document_report()` produces a valid VPAT 2.x ACR HTML/JSON report (reusing `compliance_report.DocumentReport`) for a sample corpus, and the report's `conformance` field only ever reads `Conformant`/`Partially Conformant`/`Not Conformant` — never a bespoke Office-only value — so a mixed PDF+Office compliance dashboard can aggregate both without special-casing.
6. **AC6.** Zero new runtime dependencies added to `pyproject.toml`/`requirements.txt` (NFR3 verified by `git diff` on the dependency manifest in CI).
7. **AC7 (deadline-linked).** By the Phase 1 exit date (§9), docx coverage (title/language/heading/alt-text/table-header rules — `OOXML-DOCX-{1.1,1.2,2.1,2.2,2.3,3.1,3.2,4.1}`) is at parity-or-better than the current 5-check `office_acceptance.py` baseline, measured as: same 5 existing rule_ids still pass/fail identically on the existing test corpus (no regression), plus ≥6 new rule_ids from §4.1 now evaluated (`2.2`, `2.3`, `3.2`, `4.2`, `5.1`, `6.1`).

## 10. Risks and mitigations (specific to this project's own findings, not generic)

| Risk | Specific to this project because... | Mitigation |
|---|---|---|
| **The "documentation/code mismatch" failure mode recurs internally.** `Community-Access/accessibility-agents` claimed 46 checks, shipped ~10-12 — the exact failure mode of a rule catalog drifting from its implementation. | This PRD itself lists 24 rule IDs in §4; nothing stops the same drift happening here as rules get descoped during implementation. | AC1/AC2 make catalog-vs-implementation parity a CI-enforced fact (every listed `rule_id` must have a passing+failing unit test), not a documentation claim. Any rule removed from scope must be removed from §4's table in the same PR, not left to silently rot. |
| **`office_acceptance.py`'s alt-text check only walks `doc.inline_shapes`, missing anchored/floating images.** Found while reading the actual `_check_docx` implementation this session (§4.1, `OOXML-DOCX-3.1` note) — `python-docx`'s `inline_shapes` API does not enumerate `wp:anchor`-positioned (floating/wrapped) images at all. | This is a real, already-present gap in the code being extended, not a hypothetical. A District form with a floating logo or watermark image currently gets a false "Passed" on alt-text today. | Rule `OOXML-DOCX-3.1` explicitly requires reading `w:drawing//wp:anchor/wp:docPr` in addition to `wp:inline/wp:docPr` — call this out as a Phase 1 must-fix, not deferred, since it's a correctness bug in the *existing* baseline the new rule inherits. |
| **The two existing-but-disconnected Office capability layers (deterministic `office_acceptance.py` vs. LLM `quality_judges/office/`) could get built into three, not merged into one ladder.** Both already exist independently (§1.3); a naive implementation of this PRD could add office-verify as a *third* uncoordinated layer instead of the unifying L0-L5 gate described in §3.1. | Unlike the PDF side, where `levels.py` already cleanly composes `pdf_acceptance` + `quality_result` into one `classify_level()`, Office has no such composition today, so there's no existing pattern to copy-paste — it has to be designed fresh in this PRD (§3.1 L4 row) and could be gotten wrong. | FR3 explicitly requires `office_levels.classify_level()`'s L4 gate to call `audit_office_quality().overall_pass` (not reimplement quality judging), and AC5 requires the shared `DocumentReport`/`Conformance` vocabulary — both are concrete, reviewable acceptance criteria specifically aimed at this failure mode. |
| **pptx reading-order and xlsx merged-cell/named-range rules have zero remediator support today** (FR6 gap list). | Shipping a rule that reports `Failed, fixable=True` when `office_remediator.py` has no code path to actually fix it produces a false promise to whatever consumes the report (e.g., an auto-retry loop expecting the next remediation pass to clear the failure). | FR6 explicitly enumerates the four rules with no current remediator support and requires either (a) matching remediator work lands in the same phase, or (b) the rule ships with `fixable: False` until remediator support exists — never `fixable: True` without a real code path, mirroring how the PDF side's `is_font_clause_only()` in `levels.py` explicitly marks font-clause violations as needing a different remediation tier rather than falsely promising the structural/LLM loop can fix them. |
| **Color-only-meaning and manual-list-vs-real-list rules are inherently false-positive-prone pattern matches**, not clean structural facts like `w:tblHeader` presence. | These are the OOXML rules with the weakest evidentiary basis in this PRD (§4, marked "best-effort" / "Manual Check Needed" routing) — shipping them as hard Fails would erode trust in the whole rule catalog the way a noisy linter gets muted/ignored. | §4's tables and §5's split explicitly route these to `"Manual Check Needed"` rather than `"Failed"` in the deterministic tier — consistent with how the PDF checker already has a `"Manual Check Needed"` status precedent (`CheckResult.status`) for exactly this kind of low-confidence signal, rather than inventing a new status vocabulary. |
| **No test coverage exists today for any Office code path** (`find tests -iname "*office*"` → empty), so office-verify is being layered onto an untested foundation. | Any latent bug in `office_acceptance.py`'s existing 12 checks or `office_remediator.py`'s mutation logic could be silently baked into office-verify's "baseline" without a test ever having caught it. | NFR5 requires the *first* deliverable in Phase 0 to include tests for the *existing* `office_acceptance.py` checks before any new rule is added — establishing a regression baseline, not just testing net-new code. |

## 11. Open questions requiring a human decision

1. **Rule-ID namespace and versioning scheme.** This PRD proposes `OOXML-<FORMAT>-<Checkpoint>.<Condition>` (e.g. `OOXML-DOCX-3.1`). Should it instead mirror veraPDF's `ISO 14289-1:2014-7.21.4.1-1` style (spec-name:year-clause) even though, per §1.1, no such external spec exists to cite? A human should decide whether inventing a spec-shaped name for a taxonomy with no external standard behind it is clarifying or misleading to auditors reading a future VPAT ACR.
2. **Where does `OfficeStructureProbe` (the Office analogue of `levels.StructureProbe`) live, and does it need XMP-equivalent custom metadata** (OOXML supports custom `docProps/custom.xml` properties) **to carry a District-specific conformance-profile marker**, the way PDF's `has_uaid` reads `pdfuaid:part` from XMP? There is no OOXML equivalent of a PDF/UA identifier convention — a human should decide whether to invent one (e.g., a custom property `LACCD-A11y-Profile`) for provenance tracking, or leave L3/L4 Office documents with no equivalent machine-readable "this was verified" marker embedded in the file itself.
3. **Should `office-verify` also validate documents it did *not* remediate** (i.e., campus-authored files never touched by `office_remediator.py`) as a standalone linter, and if so, does that change the CLI's threat model (arbitrary untrusted OOXML input, needing hardened zip-bomb/decompression-bomb guards beyond what `office_routes.py`'s existing size caps provide)? This PRD assumes yes (G7/FR7's CLI is format-agnostic to origin) but a human should confirm scope and any additional file-safety hardening budget.
4. **Priority of the four FR6 remediator gaps** (pptx duplicate-title dedup, xlsx default-sheet-rename, xlsx merged-cell splitting, docx merged-header-cell splitting) **relative to the Phase 1-3 schedule** — should remediator work land strictly inside the same-quarter phase as its corresponding rule, or can rules ship first (as `fixable: False`, flagged for human/queue review) with remediator support following in a later phase? §9's phase table assumes the former; a human sponsor should confirm the schedule can absorb doubled scope per phase or explicitly accept the latter, slower-remediation-catch-up path.
5. **Ownership of the "Manual Check Needed" pattern-based rules' false-positive rate.** Since color-only-meaning and manual-list detection (§4, §5, §10) are acknowledged best-effort, who reviews/tunes their precision after Phase 1-3 ships, and what false-positive-rate threshold triggers pulling a rule back to informational-only (no status impact) versus keeping it in the "Manual Check Needed" tier? Needs a named owner, not left implicit.

### B. External evidence (verified this session via primary source, cited as given — not re-derived)

- **Microsoft Accessibility Checker** — UI-only feature; no API/CLI/OSS engine exists to script it.
- **LibreOffice UNO accessibility API** — exposes the runtime AT tree of a *live, open* document for screen-reader consumption; not a static-file linter, wrong object model for this use case.
- **`odfvalidator` / `office-o-tron`** — real, JAR-based, scriptable, but validate ISO/IEC 29500/ODF XML schema and structural conformance, not accessibility semantics (alt-text, heading structure, reading order are entirely out of their scope).
- **Apache POI** — general-purpose OOXML read/write library, no accessibility module.
- **DAISY Consortium's Ace** — deterministic and real, but EPUB-only; no OOXML equivalent exists and none was found planned.
- **ISO/IEC 29500** — defines the OOXML file format itself; unlike PDF (which layers PDF/UA-1 as a WCAG-mapped conformance profile on ISO 32000), no equivalent WCAG-mapped accessibility conformance profile exists for OOXML.
- **`Community-Access/accessibility-agents`** (MIT, v6.0.0, 348 stars) — `.github/scripts/office-a11y-scan.mjs` (307 lines) read directly: genuinely deterministic (pure Node.js, `node:fs`/`node:zlib`, zero dependencies, hand-rolled ZIP central-directory parser + regex XML extraction, SARIF 2.1.0 + GitHub Actions annotation output), but implements only ~10-12 checks total across docx/pptx/xlsx versus 46 claimed in its own docs (confirmed documentation/code mismatch by reading the source), and has zero remediation capability — "fixing" there is live LLM code-generation with human review, not a deterministic library function. Architecture pattern (ZIP-central-directory + XML-regex-extraction + SARIF/CI-annotation format) referenced as a reasonable pattern, not copied as code or taken as a dependency.
- Full citation and methodology trail for all of the above: `RESEARCH_remedy_server_ADA_refactor.md` §9 ("Open-source alternatives assessment") and §9.1, item 5 ("Office-format (docx/pptx/xlsx) accessibility validator — confirmed real, industry-wide gap").
